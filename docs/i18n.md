# 閱讀存摺 i18n 導入筆記

## 一、決策與方向
- **為什麼導入**：作為作品集展示 + 未來發展成 App 時可能拓展多語。屬於「真需求」而非過度設計，值得早導入（retrofit 成本高）。
- **技術棧**：`@nuxtjs/i18n` v10 + Nuxt 4。
- **key 流派選擇**：採**結構化 key**（`t('navbar.bookshelf')`）而非自然語言 key。
  - 結構化 key 較能 scale、文案改字不動 key，缺點是 code 裡看不到中文。
  - 用 **i18n Ally** VS Code 擴充補回可讀性（在 key 旁 inline 顯示中文），locales 路徑指到 `i18n/locales`。

## 二、`$t` / `t` 的用法（依情境不同）
| 情境 | 寫法 |
|---|---|
| `.vue` `<template>` | 直接 `{{ $t('...') }}`，免 import |
| `.vue` `<script setup>` | `const { t } = useI18n()` → `t('...')` |
| 純 `.ts` / composable | `useI18n()`（setup 生命週期內）或 `useNuxtApp().$i18n.t` |
| Nitro server | 另一套機制；本專案 server 極薄，通常用不到 |

## 三、URL 策略與 SEO
- **策略選 `prefix_except_default`**（預設 zh-TW 無前綴，英文為 `/en/...`）：網址乾淨、各語言可獨立索引。
- **SEO 只對「公開閱讀主頁」有意義**；登入後私人頁不需索引（甚至該 `noindex`）。
- `@nuxtjs/i18n` 會自動生成 `hreflang`、`og:locale`、canonical（啟用 `useLocaleHead()`）。
- `detectBrowserLanguage.redirectOn: 'root'` 比 `'all'` 安全：深層分享連結不會被自動改語言導走。

## 四、Vue 響應式踩雷（重要基礎）
- **`t()` 的結果放進普通 `const` 會被「快照」凍住**，切語言不更新。
- 凡是「從 `t()` / `locale` / route 衍生」的資料（選單、表格欄位、麵包屑…），都要用 **`computed`**：
  ```ts
  const items = computed<NavigationMenuItem[]>(() => [ /* ...t()... */ ])
  ```
- 語言切換器顯示 active：比對 `l.code === locale`（取 `const { locale } = useI18n()`），**不要靠 `router-link-active`**（前綴策略下路由比對不準）。迴圈變數別叫 `locale` 以免遮蔽。

## 五、Supabase × i18n 路由衝突

### 5-1 登入/回跳頁的語系（2026-07-17 改版）
> ⚠️ 舊版曾用 `defineI18nRoute(false)` 鎖住 login/confirm，**已廢棄**。原因見下。

**踩過的坑**：`login.vue`、`confirm.vue` 原本加 `defineI18nRoute(false)`，想對齊 Supabase 寫死的 `/login`、`/confirm`。但這等於把兩頁**鎖在預設語系**：
- 英文使用者重整 `/login`、或 OAuth 整頁回跳後，語系被重置成繁中。
- 語系只能靠 `i18n_redirected` cookie 傳遞，而該 cookie 落在**無前綴/被排除路由**時會被 i18n 覆寫成預設語系 → **不可靠，勿當語系載體**。
- 根因：`defineI18nRoute(false)` 的語意就是「這頁退出 i18n 路由系統」，連帶關掉瀏覽器偵測。用它來處理「要跟隨使用者語言」的頁是選錯工具。

**現行做法（讓前綴路由承載語系）**：
- login/confirm 改回**一般本地化路由**（`/en/login`、`/en/confirm` 都存在），移除 `defineI18nRoute(false)`，改補 `noindex`（認證頁本就不需索引）。
- `detectBrowserLanguage.redirectOn` 由 `root` 改 **`no prefix`**：讓無前綴路由（`/login`）也依瀏覽器語系自動導向前綴版。
- OAuth `redirectTo` 用 `` `${origin}${localePath('/confirm')}` `` 帶回對應語系。
- `useLoginRedirect` 簡化為 **URL 驅動**（`navigateTo(localePath('index'))`），不再讀 cookie。
- `config/routing.ts` 的 `authPaths` 前綴版納入 `excludePaths`，避免未登入者進 `/en/login`、`/en/confirm` 被守衛踢造成**迴圈**（見第六節）。
- 已移除無作用的 `skipSettingLocaleOnNavigate` 實驗。

**關鍵認知**：`@nuxtjs/supabase` 的 `redirectOptions.callback`（`/confirm`）**沒有 code 交換魔法**——看 `auth-redirect.js`，它只是被拿去當路由守衛的放行 pattern（`[login, callback, ...exclude]`）。真正的 code 交換是 supabase-js `detectSessionInUrl`（預設開）在 client 端做的，不看路徑。所以 callback 帶不帶語系前綴，模組完全不在乎，`/en/confirm?code=...` 一樣正常換 session。

### 5-2 公開頁與白名單
- **公開頁要放行兩版**：`prefix_except_default` 下，`exclude: ['/']` 擋不到英文首頁 `/en`，英文訪客會被登入 middleware 攔截。每個公開頁都要放行 zh + `/en` 兩條路徑。
- **不要用萬用 glob**（如 `/*/u/*`）偷懶放行受保護頁：語意太寬、易誤放行。授權邊界要明確可預測。
- **Supabase 後台 Redirect URLs** 例外可用 wildcard：`/*/confirm`（`*` 只吃單一路徑段，locale code 不含 `/` 剛好命中；含 dev localhost 各一組）。這是 repo 外唯一手動設定，**加語言時不用回來改**，但部署前要確認正式站也加了。`/**`（放行整個網域）範圍過寬、不建議。

## 六、可維護的 exclude 架構（單一事實來源）
- 抽出 `config/routing.ts`（純 TS、不碰 Nuxt runtime），由 `locales` + `publicPaths` + `authPaths` **自動衍生** exclude：
  ```ts
  const prefixedLocales = locales.map(l => l.code).filter(c => c !== defaultLocale)
  const prefixed = (path: string) =>
    prefixedLocales.map(code => (path === '/' ? `/${code}` : `/${code}${path}`))

  export const publicPaths = ['/', '/u/*']      // 免登入公開頁
  export const authPaths = ['/login', '/confirm'] // 認證流程頁（要從守衛放行避免迴圈）

  export const excludePaths = [
    ...publicPaths.flatMap(path => [path, ...prefixed(path)]),
    ...authPaths.flatMap(path => [path, ...prefixed(path)]),
  ]
  ```
- **加語言只改 `locales`、加公開頁只改 `publicPaths`、加認證頁只改 `authPaths`**，exclude 不用重寫。
- `authPaths` 的無前綴版（`/login`、`/confirm`）其實模組會自動放行（`login`/`callback`），但明列不衝突、較好讀；真正**必要**的是把 `/en/login`、`/en/confirm` 納入，否則會迴圈。
- `nuxt.config.ts` 用**相對路徑** import（`./config/routing`，別用 `~`/`@` 別名）。
- `config/` 資料夾**不與 Nuxt 4 慣例衝突**（不在保留清單、在 `srcDir` `app/` 外）。

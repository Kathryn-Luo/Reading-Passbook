# ADR-005：等級與勳章規則做成資料驅動，不寫死在程式

- 狀態：已採納
- 日期：2026-07-07

## 背景

產品有三種累積回饋：等級（經驗值主線）、勳章（離散支線成就）、書幣。
其中等級門檻與稱號、勳章的觸發條件，都是「上線後幾乎一定會想調」的東西
——某條等級曲線太陡、某個勳章太好拿、想新增稱號或成就。

另外，等級的經驗值來源刻意避開主觀的「難易度」（誰來定難易度都有爭議），
改用三個客觀可算的訊號：本數、類型廣度、筆記投入。這三個的權重也會想調。

## 決策

把等級門檻、稱號、勳章觸發條件都做成**資料驅動 (data-driven)**——
存成設定檔或資料表，而非寫死在程式邏輯裡。

- 等級門檻是一張對照表：`[{ level, min_exp, title }, ...]`，
  計算等級 = 用總經驗值查這張表。
- 經驗值權重（本數 +100 / 新類型 +50 / 筆記 +20）放設定，可調。
- 勳章是一張規則清單：每個勳章一組觸發條件，結算時逐條檢查。

新增一個勳章、調一條曲線、改一個稱號 = 改資料，不改程式、不重部署。

另有一條相關原則：**等級解鎖「表達」而非「功能」**——高等級解鎖的是
封面樣式、帳本紙質、公開頁主題、稱號、更深的統計圖，而不是把某些功能
鎖起來。理由見下。

## 為什麼不寫死在程式

把 `if exp >= 5400: level = 12` 這類邏輯寫進程式最直接，但：

- 每次調整都要改 code、跑一次部署，迭代慢。gamification 的數值本來就
  需要反覆微調，寫死等於每次微調都是一次工程變更。
- 規則散在程式各處，難以一眼看出「目前所有等級/勳章長怎樣」。

資料驅動則把「規則」和「執行規則的引擎」分開：引擎穩定不動，規則是資料、
隨時可調。這對一個會長期微調數值的系統特別重要。

## 為什麼等級解鎖「表達」而非「功能」

若高等級才解鎖某些功能，等於給新手一個殘缺的 App——這是在懲罰新手，
對留存是負的。改為解鎖「表達」（外觀、稱號、統計深度），則：
- 新手不會有功能被剝奪的挫折。
- 老用戶得到身份展示的空間，而這些外觀/稱號又餵養公開頁與社群
  （別人點進你的公開頁，一眼看到等級與稱號 = 社交貨幣）。

## 後果

**好處**
- 數值可快速迭代：改資料不改 code、不重部署。
- 規則集中、一目了然，易於維護與擴充。
- 「解鎖表達不解鎖功能」保護新手體驗，同時強化社群展示。

**代價 / 取捨**
- 需要先設計好規則的資料結構與結算引擎，前期比寫死多一點成本。
  → 一次性投入，之後每次調整都省，長期划算。
- 資料驅動的規則若設計不良（欄位不夠通用），新型態的勳章仍可能需要
  改引擎。第一版先涵蓋已知的勳章型態即可，不過度預設。

## 備註
經驗值本身存成流水（`exp_delta` 加總，見 ADR-001），與等級引擎分離。
這也留了門：未來 App 版的寵物成長階段，本質是同一份經驗值的另一種外皮，
共用來源、不需另建系統。

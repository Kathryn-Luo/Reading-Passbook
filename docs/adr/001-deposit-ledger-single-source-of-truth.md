# ADR-001：以「存入流水」為單一真相，衍生值全部用算的

- 狀態：已採納
- 日期：2026-07-07
- 相關：ADR-000（重寫決策）、後續 ADR-002（沖正刪除）、ADR-003（本數去重）

## 背景

閱讀存摺要呈現的累積數值有一整組：累積本數、累積頁數、書幣餘額、經驗值、等級、勳章進度、每一筆帳列的「結餘」。同時，產品有三個會動到這些數值的功能：

1. **分批存入**：一本書可以一章一章或一段頁數存入，最後才讀完。同一本書會有多筆存入紀錄
2. **刪除/編輯**：使用者可能存錯、手滑、或是讀到一半棄書想移除
3. **重讀**：同一本書可以再讀一次，要記錄行為，但不該重複計算累積本數

設計資料核心時的核心選擇是：這些累積數值要「直接存一個欄位」還是從一張存入流水表即時算出來


## 決策

建立一張 `deposits` 存入流水表作為「單一真相」。每一次「存入存摺」的動作 = 一筆 deposit 紀錄，記錄這筆貢獻的頁數、書幣、經驗值，以及它的類型（首讀進度/首次讀完/重讀/沖正）

所有累積數值都是從**這張表推導的衍生值**，不存快照、不手動加減：
| 衍生值 | 算法 |
| ----- | --- |
| 累積本數 | `COUNT(DISTINCT book_id) WHERE counts_as_book  = true` |
| 累積頁數 | `SUM(pages_delta)` |
| 書幣餘額 | `SUM(coins_delta)` |
| 總經驗值 | `SUM(exp_delta)` |
| 帳列結餘 | 依時間排序的 running total (window function) |

（全部帶 `WHERE is_voided = false`，排除已沖正的紀錄）


## 為什麼不存 total 快照
存快照（每次存入就去改一個 `total` 欄位）是直覺、也是效能最好的做法，但他在下面三個功能都會出問題，而且問題會互相疊加：
- 分批存入：同一本書多筆存入，「這本書算幾本、進度到哪」要靠加減維護，一旦某次寫入失敗或重試，數字就對不上
- 刪除：要記得把當初加上去的 total 減回來，包含頁數、書幣、經驗值、可能還有已觸發的勳章。任何一處漏減，帳就永遠歪掉，而且事後無從稽核
- 重讀：要寫額外邏輯判斷「這本書之前是不是已經算過本數了」，才能避免重讀灌水。判斷邏輯散在各處，容易漏

這三個「加減 total」的地方，只要有一個寫錯，數字就對不起來，而且**因為沒有流水，錯了也查不出哪裡錯**

用流水表則反過來：這三個功能全部收斂成「寫一筆分錄」，數值永遠從流水重新算，不存在「忘了減回來」的可能。特別是防重讀灌水——本數用`COUNT(DISTINCT book_id) WHERE counts_as_book`，而重讀紀錄的 `counts_as_book = false`，所以 distinct 本數不會變，不需要任何額外的「檢查是否重讀」邏輯。規則被編碼進 schema，而不是散在程式裡。

## 後果

**好處**
- 一致性有保障：帳永遠對得起來，因為只有一個真相來源
- 三個功能共用一個寫法（寫紀錄），程式簡單，不易出錯
- 完整可稽核：每一筆存入、沖正都留痕跡，能重建任何時間點的狀態
- 前端的存摺「帳列」畫面，直接就是這張流水表的呈現，資料結構與 UI 天然對齊

**代價 / 取捨**
- 每次查詢都要 `SUM` / `COUNT DISTINCT` ，理論上比讀一個欄位慢。
  → 第一版本資料量小，即時計算也夠快，先不改善
- 帳列結餘要用 window function 算 running total，比存一個欄位複雜

**效能對策(先不做，留給之後)**
若後資料量大到即時計算變慢，再加一張物化快取表（如 `user_stats`），用資料庫觸發器或定時任務從流水重算。快取永遠可以從流水重建，流水仍是唯一真相。先做即時計算，簡單且不會出錯。過早加快取只會帶來「快取與流水不同步」的 bug


## 備註
這個「事件流水 + 衍生狀態」的模式，本質上是把會計帳本（appended-only、用沖正而非塗改）的概念套進閱讀紀錄。刪除的沖正機制另外見 ADR-002。本數的去重計數另見 ADR-003。三者其實是同一個設計哲學的三個面向。

# ADR-003：「本數」算讀過幾本不同的書，不算讀完幾次

- 狀態：已採納
- 日期：2026-07-07
- 相關：ADR-001（存入流水為單一真相）


## 背景
「累積本數」是這個產品最核心的數字，存摺餘額、等級、勳章都建立在它之上。
但「重讀」這個功能帶來一個定義問題：同一本書讀第二次，本數該不該再 +1 ?


如果會 +1，就出現一個刷分漏洞：把同一本書讀十遍，就有十本的累積、十倍的書幣。存摺、等級、勳章全部失去意義。
但如果重讀就把之前的歸零，是變相懲罰使用者，還違背了「餘額只增不減」的核心

## 決策

把「**書籍**」和「**閱讀**」的行為分開，明確定義：
>  **本數 = 讀過幾本「不同」的書（廣度），不是讀完幾「次」**

實作上：
- 一本書的「首次讀完」記一筆 `kind = finish_first`, `counts_as_book = true`  
  並設定 `books.first_finished_at` （一生只設定一次，重讀不復寫）
- 重讀記一筆 `kind = reread`, `counts_as_book = false`
- 累積本數 = `COUNT(DISTINCT book_id) WHERE counts_as_book = true`

重讀的行為仍被記錄、仍有價值：可累積閱讀天數、頁數，可觸發重讀的這類勳章

## 為什麼用 DISTINCT 而不是額外邏輯
另一種做法是：允許重讀也記錄 `counts_as_book`，但在存入時寫程式判斷「這本書之前是不是已經算過了」，算過就不再增加

不選擇這條路，因為：
- 判斷邏輯會散在存入流程裡，是額外的、容易漏的特例
- 一但有併發或重試，「之前算過沒」的判斷可能出錯

用 `COUNT(DISTINCT book_id) WHERE counts_as_book` 則把規則編碼進查詢本身，重讀分錄的 `counts_as_book = false` ，根本不進入計數；就算某本書有一筆 finish_first + 五筆 reread，distinct 後仍是一本，不需要任何「檢查是否重讀」的程式邏輯，資料模型本身就防住了灌水

這是「把規則放進 schema ，而不是放進程式」的典型例子——規則只有一處、不會漏、不會被繞過

## 後果

**好處**
- 累積數字誠實：本數反映真實的閱讀廣度，經得起使用者檢視
- 防灌水不靠程式判斷，靠資料模型天然保證，維護成本趨近於零
- 重讀行為仍被鼓勵（有記錄、有專屬勳章），不犧牲想重讀的使用者

**代價/取捨**
- 「讀完次數」若未來想呈現（例如「這本書你讀了 3 次」），要另外從 reread 分錄數去算，但這是加值資訊，不影響核心本數
- 需要在 UI 上讓使用者理解「重讀不加本數」，避免誤會系統沒記錄到  
  → 可在重讀存入時明確提示「已記錄重讀，本數不重複計算」

## 備註
與 ADR-001, ADR-002 同屬「事件流水」設計哲學的一環
把業務規則編碼進資料模型與查詢，而非散在應用程式碼裡

# ADR-000：打掉重練，而非漸進重構

- 狀態：已採納
- 日期：2026-07-07

## 背景

閱讀存摺三年前以 Nuxt 3 + Firebase 實作（見已封存的 read-passbook）。
這次想把它做成一個更完整的個人作品，面臨的選擇是：在舊 repo 上漸進重構，
還是打掉重練。

舊版的兩個根本限制：
1. **資料庫典範不合**。Firebase 是 NoSQL 文件資料庫，而本專案的核心
   （閱讀存入流水、從流水推導所有累積數值、公開/私密的關聯權限）
   是典型的關聯式問題。在 Firestore 上做這些需要大量 denormalize，
   且難以保證累積數值的一致性。
2. **權限散在應用層**。舊版授權邏輯寫在前端與雲函式，分散且易漏。

## 決策

打掉重練，用 Nuxt 4 + Supabase (Postgres + RLS) 重寫，並開新的乾淨 repo。

關鍵理由：
- 框架與資料庫典範都要換（Nuxt 3→4、Firebase→Postgres），
  漸進遷移的成本高於重寫。
- 重寫能一次把資料模型做對——以單一「存入流水表」為真相來源，
  本數/頁數/書幣/經驗值全部從它算出（見後續 ADR-001）。
- 權限改用 Supabase RLS 下沉到資料庫層，前端幾乎不寫授權邏輯。

## 後果

**好處**
- 資料模型從一開始就正確，避免帶著舊架構的包袱前進。
- 新 repo 的提交歷史乾淨，能清楚呈現這次的開發過程。
- 從 NoSQL 換到關聯式，是這次重寫最主要的技術收穫。

**代價 / 取捨**
- 放棄舊版既有程式碼與提交歷史（已用 archive 保留，非刪除）。
- 舊版若有既存資料，需另寫一次性搬遷（資料量小，可手動處理）。

## 備註
舊 repo 已設為 archived（唯讀、明確標示舊版），並在其說明導向此新專案，
保留「同一想法、三年後用更好架構重寫」的演進痕跡。

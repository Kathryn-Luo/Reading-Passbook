# ADR-002：刪除用「沖正分錄」

- 狀態：已採納
- 日期：2026-07-07

## 背景
使用者需要能刪除或修改存入紀錄，存錯書、手滑重複存、讀到一半棄書想清掉，都是真實會發生的情況。所以需要有「刪除/編輯」的功能。

ADR-001 決定了所有累積數值（本數、頁數、書幣、經驗值、等級、勳章）都從 `deposits` 流水計算，「刪除一筆存入」不只是刪除一列資料，還會牽動一連串已經發出去的獎勵。


## 決策

刪除不做硬刪除。刪除 = 新增一筆 `kind = void` 的沖正分錄：
- 沖正分錄的 `pages_delta`, `coins_delta`, `exp_delta` 為原分錄的負值
- `reverses_deposits_id` 指向被沖正的原分錄
- 原分錄標記 `is_voided = true`
衍生值計算一律帶 `WHERE is_voided = false`，被沖正的分錄自然被排除，餘額、書幣、本數全部自動回正。原始資料保持不動。

「編輯」則視為「沖正舊的 + 新增正確的」兩筆分錄組合


## 為什麼不硬刪
硬刪（`DELETE FROM deposits WHERE id = ...`）看似乾淨，但會踩到幾個雷：
- 已發出的獎勵收不回：
  刪掉一筆 `finish_first`，當初給的書幣、經驗值、可能觸發的勳章都要手動逆向處理。漏一個，帳就歪了
- 歷史結餘斷裂：
  帳列的「結餘」是依時間排序的 running total，中間硬刪一筆，該筆之後每一列的結餘都要重算，且無從得知「這裡曾有一筆被刪掉」
- 無法稽核：
  存摺這種「累積資產」的產品，使用者會信任數字。一但有無痕的硬刪，出現對不上的情況時，完全查不出原因，玩家只會覺得「系統作弊/偷錢/被鎖帳」，信任蕩然無存
- first_finish_at 難處理：
  若硬刪的是某一本書唯一的 `finish_first` 分錄，這本書該不該算「讀過」變得模糊，狀態難以正確回退

沖正法把「刪除」也變成一筆「寫一筆分錄」，跟 ADR-001 的其他操作完全同構，而天然可稽核，任何時間點的狀態都能從流水重建。


## 後果
**好處**
- 帳永遠對得起來，且完整可稽核（append-only，只增不改）
- 刪除、編輯、存入 共用同一種操作（寫入分錄），程式一致不易出錯
- 對使用者體驗上就是「刪掉了」，感受不到底層的沖正機制


**代價/取捨**
- 資料只增不減， `deposits` 表會累積被沖正的分錄。  
（初期專案資料量小，可忽略。日後可加封存機制）
- 查詢一律要帶 `is_voided = false`，需在每個衍生值查詢中一致套用（建議包成共用的 query helper，避免漏加）


## 備註
這是會計帳本的基本原則：
帳本用紅字沖銷，不用塗改。把它套進閱讀紀錄，換來的是「累積資產」該有的可信度。
與 ADR-001, ADR-003 同屬「事件流水」設計哲學的一環

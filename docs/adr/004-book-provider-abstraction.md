# ADR-004：書籍資料源抽象成 Provider，Google Books 為主、手動為備援

- 狀態：已採納
- 日期：2026-07-07

## 背景

加一本書時需要書目資料（書名、作者、封面、頁數、ISBN）。使用者是台灣
讀者，理想上希望台灣的書（含繁中書）都查得到。

調查後的現實：**博客來、讀墨（Readmoo）都沒有公開的官方書籍 API**。
網路上流傳的「博客來 API」多是第三方爬蟲包出來的非官方接口，不穩定、
隨時失效、也有服務條款風險。把產品的核心資料源綁在爬蟲上並不妥當。

## 決策

不直接綁定任何單一資料源，而是定義一個 `BookProvider` 介面，底下可掛
多種實作，並排定優先序：

1. **Google Books API（主力）** — 官方、免費、支援 ISBN 查詢，
   對台灣書與繁中 metadata 涵蓋度堪用。透過 Nitro server route
   `/api/books` 代理（避免露 key、繞 CORS）。
2. **手動輸入（備援）** — 查不到的書，讓使用者自行填書名/作者/頁數。
   這不是退路，是必要的 fallback。
3. **（未來可擴充）** 封面補圖服務、國家圖書館 ISBN 書目等，
   都只是新增一個 provider 實作，不動既有程式。

## 為什麼抽象成介面，而不是直接串 Google Books

直接在程式各處呼叫 Google Books 最快，但會把整個 App 綁死在單一外部
服務上。抽象成 provider 的理由：

- **資料源會變**：今天用 Google Books，未來可能想補台灣書源、或某服務
  改版/收費。介面隔離讓「換來源」只影響一個實作檔，不擴散。
- **fallback 是常態**：查不到書必然發生，手動輸入和 API 查詢應該是
  同一個介面下的兩種 provider，而非散落的 if-else。
- **可測試**：介面能用 mock provider 測試上層邏輯，不必真的打外部 API。

## 後果

**好處**
- 資料源可替換、可組合（主力 + 備援 + 未來擴充），不綁死單一服務。
- 「台灣書源」成為明確的未來擴充點，寫進 roadmap 即可，不阻塞第一版。
- 是「依賴反轉 / 介面隔離」的實際應用，架構上乾淨。

**代價 / 取捨**
- 比直接串 API 多一層抽象，第一版只有一個真正的 provider（Google Books）
  時，介面看起來像過度設計。
  → 但因為 fallback（手動輸入）本來就存在，介面至少一開始就有兩個
    實作，不算純為未來過度抽象。

## 備註
明確不採用博客來/讀墨爬蟲作為主資料源。若未來要接台灣書源，優先考慮
有正式書目資料的管道（如國家圖書館 ISBN 網），並以新 provider 形式加入。
